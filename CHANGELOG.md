# Changelog

Все заметные изменения Beacon. Формат — [Keep a Changelog](https://keepachangelog.com/),
версионирование — [SemVer](https://semver.org/). Политика бампа: `docs/versioning.md`.

Версия — единый источник: `package.json` (`manifest.config.ts` читает `pkg.version`).
До `1.0.0`: каждая **новая фича/модуль** (live-verified) = MINOR `0.x.0`; багфикс/UX/honesty/
рефактор/debt = PATCH `0.x.y`. `1.0.0` = весь план Todoist закрыт, нет заглушек/хардкода,
грабли учтены, E2E зелёный.

## [0.11.1] — 2026-07-21

### Fixed
- **PYMK-добор теперь срабатывает при ЧАСТИЧНОМ поиске, а не только при нулевом.**
  При лимите поиска LinkedIn страница отдаёт 1-2 коннектных → пасс делал 2 коннекта и вставал,
  хотя дневной кап ~14 — приходилось запускать прогон второй раз, чтобы добрать из PYMK. Гейт
  фолбэка (`runConnectWithFallback`) был «всё-или-ничего» (`search.executed > 0` → возврат);
  теперь добор идёт по недобору run-cap (`search.capReached`). Один прогон добирает дневной
  лимит: `executed = search + pymk`, дневной кап соблюдён (PYMK перечитывает уменьшенный бюджет →
  `search + pymk ≤ dailyCap`). Намеренные пути (`no_keywords`/`empty_search`/`pool_pending` → PYMK;
  `disabled`/`budget`/`cancelled` → стоп) сохранены.

## [0.8.4] — 2026-07-10

### Fixed
- **Несколько комментариев (до 5) под одним постом — разный текст, за один прогон.**
  `findSubmit` шёл вверх на 8 уровней и отбрасывал `disabled`-кнопки; submit своего поста
  коммитится асинхронно (ProseMirror) и пару тиков `disabled` → walk проскакивал свой
  comment-box и на общем предке ленты хватал **включённый** submit ЧУЖОГО открытого
  редактора → коммент постился под чужой пост, разные комменты копились на одном. Фикс:
  `findSubmit` останавливается на первом предке со submit-кнопкой (свой box, считая disabled)
  и ждёт СВОЙ submit; иначе коммента нет. Инвариант: коммент уходит под правильный пост или
  никуда — никогда под чужой. Покрыто юнит-тестами (escape к соседнему box'у).

## [0.8.3] — 2026-07-09

### Fixed
- **Коннекты + Просмотры молча ничего не делали, хотя поле поиска выглядело заполненным.**
  Карточка «Модули» показывала prefill ключевых слов из экспертизы, но **не сохраняла** его
  (`v-model` пишет только по `@change`) → прогон читал пустой `connect:settings.searchKeywords`
  → `no_keywords` → оба модуля no-op. `ensureSearchKeywords` персистит prefill при открытии
  карточки (регионы сохраняются); прогон по-прежнему читает только сохранённое — реально не
  настроенный модуль остаётся безопасным no-op с честным «не задан поиск» (без авто-коннекта
  по выдуманному дефолту, safety-first).
- **Идеи для постов не извлекались в прогоне (мёртвый catch-up).** Регрессия с `c4981e5`
  (28 июня): run-end извлечение идей было под гейтом `autopilotRunning`, который `endRun()`
  зануляет ДО `finally` → catch-up никогда не срабатывал; идеи шли только mid-run путём при
  ≥25 уникальных постах (`IDEA_TARGET`), редко достижимо. Введён loop-local `extractAtEnd`
  (ставится синхронно на естественном конце: feed_exhausted / budget / risk), user-STOP
  пропускается (без повторного зажигания оверлея). Live-verified через CDP (флип `ideas:lastRun`
  на сегодняшнюю дату после feed-exhausted прогона).

## [0.8.1] — 2026-07-03

### Fixed
- **Провайдер Gemini переставал генерировать (идеи/комменты), OpenRouter работал.** Причина
  (заземлено на реальном аккаунте): прямой Gemini free-tier = **5 запросов/мин** + периодический
  `503` overload; один прогон бьёт больше вызовов подряд → `429 RESOURCE_EXHAUSTED`/`503` → шаг
  молча падал (OpenRouter не страдает — пул). Добавлен **retry с backoff** для транзиентных ошибок
  (429/5xx), уважающий server-advised `retryDelay` (Gemini «retry in Xs»), с капом (30с) и **честной
  ошибкой** при реальном исчерпании квоты вместо тишины. `RetryingLlmProvider` оборачивает все
  провайдеры (`createLlmProvider`); `FetchHttpClient` бросает типизированный `HttpError{status,
  retryAfterMs}`. Изолированные операции (одна генерация) теперь надёжны; полноценный прогон на
  free-tier gemini-2.5-flash всё равно упирается в 5/мин (лимит Google) — для прогонов лучше модель
  с бóльшим RPM (2.0-flash-lite / 2.5-flash-lite) или OpenRouter.

## [0.6.1] — 2026-06-29

### Fixed
- **Само-вовлечение: бот лайкал и комментил (×3) СОБСТВЕННЫЙ пост.** Авто-опубликованный пост висел
  вверху ленты, а `LikeFilter` не знал владельца. Теперь `readOwnerName` читает владельца из ленты
  (vanity = первый `/in/` self-card → имя из непустого `alt` аватара, verified live), и движок
  пропускает посты, где автор = владелец (`own_post`). Матч по ИМЕНИ иммунен к churn componentkey —
  причине тройного коммента. При неудаче опознания — fail-open + warn.
- **Просмотр профилей делал 0–3 из 40 после серии коннектов.** Views переиспользовал connect-харвестер
  (только кнопка «Invite to connect»), поэтому уже приглашённые («Pending») были невидимы → harvest 0 →
  пагинация не стартовала. Новый `harvestProfiles` берёт людей по member-componentkey (connect + pending),
  тот же numeric memberId (seen-set валиден). Live: 3 → **38 просмотров**, 35 новых уникальных профилей.

### Changed
- **Просмотры листают страницы вглубь до капа СВЕЖИХ.** `harvestPeoplePaginated` получил `isFresh`-предикат:
  считает невиденных к таргету и листает (до 20 страниц), пока не наберёт `cap` непросмотренных или поиск
  не кончится. Честный reason `pool_dry` («свежих профилей меньше лимита») вместо молчаливого `done`.
  Smart Connect не затронут (по-прежнему только коннектабельные). Убран мёртвый `HARVEST_PEOPLE`.
