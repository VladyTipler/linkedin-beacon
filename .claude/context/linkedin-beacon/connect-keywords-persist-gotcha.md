# Gotcha — Connect/Views search keywords: display-vs-persist mismatch (2026-07-09)

## Симптом
Прогон: **Лайки работают, а Коннекты + Просмотры молча ничего не делают** (Посты «не день публикации» — норма). В отчёте: `Коннекты 0 — «не задан поиск»`, `Просмотры 0 — «не задан поиск»` (`no_keywords`). «Вчера работало, код не трогал».

## Корень
Коннекты и Просмотры делят одну настройку — `connect:settings.searchKeywords` (оба через `loadConnectSettings`). Лайки её НЕ используют → работают (это и разводит LLM/DOM: мёртвая тройка navigate+harvest, ноль LLM).

`ModulesScreen.vue` `onMounted`: если сохранённых ключей нет — подставлял в поле **дефолт из экспертизы** (`defaultConnectKeywords`) и **НЕ сохранял** его (`v-model` пишет только по `@change`). Итог: поле ВЫГЛЯДИТ заполненным («всё задано»), но в хранилище пусто → SW читает `''` → `no_keywords` → no-op.

**Почему «вчера работало»:** `chrome.storage.local` расширения обнулился (почти наверняка при переподключении/переустановке unpacked под свежую сборку v0.8.x — удаление unpacked затирает storage). Причину wipe эмпирически НЕ подтверждали — фикс закаляет против симптома. Тот же класс, что прошлый «0 коннектов = stale dist» (obs 11092).

## Диагностика (решает мгновенно)
Отчёт прогона показывает reason по каждому модулю (`reasonLabels.ts`). `no_keywords` = «не задан поиск» разводит «LinkedIn сменил DOM» и «настройки пусты» без гадания. Дамп: в SW-консоли `chrome.storage.local.get('connect:settings')`.

## Фикс (v0.8.x) — БЕЗОПАСНЫЙ вариант (выбор Влада)
`ensureSearchKeywords(store)` в `lib/connect/settings.ts`: возвращает сохранённые ключи, а если пусто — **персистит** дефолт из экспертизы (регионы сохраняет). Панель зовёт его в `onMounted` → поле больше не показывает несохранённое значение.

**SW (прогон) НЕ трогали** — читает только сохранённое: реально не настроено (карту не открывали / ключи стёрты) → безопасный no-op + честный сигнал «не задан поиск». Т.е. прогон действует ТОЛЬКО по ключам, которые юзер видел в карточке.

## Отвергнутый вариант (важно!)
Сначала сделал «агрессивный»: SW сам резолвит дефолт в рантайме → включённый модуль ВСЕГДА коннектит по выдуманному `«…recruiter»`, даже если юзер ключ не выбирал; убивало сигнал `no_keywords`. Для anti-ban/safety-first app это **молчаливая смена safety-поведения** в ban-чувствительном модуле — решение юзера, не агента. Advisor поймал. → вынес как выбор, Влад выбрал безопасный.

**Урок:** не менять молча safety-поведение (авто-действие по фабрикованному дефолту) в ban-чувствительных модулях — выносить как решение. Related: [[gotchas]], [[architecture-overview]], [[verify-the-fixed-path]].
